# 다음 단계: 모놀리식으로 다시 구현해보는 DDD 연습

이 저장소를 하루 정도 진득하게 읽고 나면 보통 두 가지 감각이 남는다.

- 도메인 경계와 상태 전이를 코드로 옮기는 방식은 매우 배울 만하다
- 하지만 이 구현이 그대로 production 완성형이라고 보기는 어렵다

바로 그 지점 때문에, 이 저장소를 읽는 다음 단계로 "직접 다시 만들어 보기"가 의미 있다.

## 왜 다시 만들어 보는가

코드를 읽는 것만으로는 DDD 를 이해했다고 보기 어렵다.
실제로 중요한 것은 아래 질문에 스스로 답하면서 경계를 자르는 경험이다.

- 왜 `Book` 과 `BookInstance` 를 나누는가
- 왜 `Patron` 이 hold 자격 판단의 중심이 되는가
- 왜 `AvailableBook`, `BookOnHold`, `CheckedOutBook` 같은 상태 객체가 필요한가
- 왜 `catalogue` 와 `lending` 을 분리하는가
- 어디까지 event 를 쓰고, 어디서는 단순한 CRUD 로 끝내는가

이 프로젝트를 직접 재구성해 보면 pattern 을 외우는 것보다 훨씬 많은 것을 배우게 된다.

## 왜 모놀리식으로 시작하는가

DDD 는 마이크로서비스를 요구하지 않는다.
오히려 학습 단계에서는 모놀리식이 더 낫다.

- 분산 시스템 복잡도보다 도메인 모델링 자체에 집중할 수 있다
- 이벤트를 프로세스 내부에서 먼저 검증할 수 있다
- bounded context 를 코드 구조로 분리하는 연습을 하기에 충분하다
- 실패 원인을 인프라보다 모델에서 먼저 찾을 수 있다

즉 목표는 "마이크로서비스 흉내"가 아니라 "도메인 경계와 상태 전이를 스스로 설계해 보는 것"이다.

## 직접 구현할 때 가져갈 것

이 저장소에서 특히 가져갈 가치가 큰 것은 아래 네 가지다.

1. `catalogue` 와 `lending` 의 분리
2. 상태를 `enum` 대신 타입으로 표현하는 방식
3. 도메인 이벤트로 다른 모델을 동기화하는 방식
4. 동일 복사본에 대한 동시 변경 충돌을 막기 위한 optimistic locking

특히 `lending.book.Book` 을 "실제 복사본 상태 aggregate" 로 보는 관점은 직접 구현하면서 반드시 체득할 필요가 있다.

## 직접 구현할 때 단순화할 것

첫 버전에서는 아래 정도로 단순화하는 것이 좋다.

- 하나의 application 안에서만 동작하는 모놀리식
- 내부 domain event 발행만 사용
- outbox, broker, saga 같은 분산 인프라 생략
- branch 는 하나로 고정하거나 최소한의 식별자만 유지
- 운영용 인증/권한/관측성보다 도메인 흐름을 먼저 구현

학습용 첫 구현에서 중요한 것은 "모든 것을 넣는 것"이 아니라 "핵심 경계를 흐리지 않는 것"이다.

## 권장 구현 범위

첫 번째 구현 범위는 아래 정도면 충분하다.

1. `catalogue`
   `Book`, `BookInstance` 등록
2. `lending`
   `AvailableBook -> BookOnHold -> CheckedOutBook -> AvailableBook`
3. `patron`
   hold 가능 여부 판단
4. `read model`
   patron profile 또는 daily sheet 중 하나

여기까지 구현해도 이미 DDD 의 핵심 질문 대부분을 직접 다루게 된다.

## 블로그에 쓰면 좋은 포인트

단순 구현기보다 아래 질문에 답하는 글이 더 좋다.

- 왜 CRUD 가 아니라 상태 전이 모델을 택했는가
- 왜 `BookInstance` 와 대출 도메인의 `Book` 을 분리해서 보았는가
- 왜 `Patron` 이 규칙을 들고 있어야 하는가
- 왜 event 기반 동기화를 썼고, 어디까지가 유효했는가
- production 으로 가기 전에 무엇이 더 필요했는가

즉 "어떻게 만들었다"보다 "왜 이렇게 나누었고 무엇이 부족했는가"를 설명하는 글이 더 가치 있다.

## production 관점에서 따로 볼 것

이 저장소는 학습용으로 훌륭하지만, production readiness 까지 모두 보여주지는 않는다.
직접 구현한 뒤에는 아래를 별도로 점검하는 것이 좋다.

- API 완결성
- 충돌 시 재시도 정책
- event 전달 실패 복구
- read model 갱신 실패 처리
- observability 와 운영 진단 가능성
- aggregate 별 정합성 전략의 일관성

이 체크리스트까지 포함하면, 단순한 따라 만들기를 넘어 설계 판단까지 정리한 문서가 된다.

## 한 문장 요약

이 저장소를 읽은 다음 단계는 "비슷한 구조를 다시 만드는 것"이 아니라,
"모놀리식 안에서 경계, 상태 전이, event, 정합성 장치를 직접 선택해 보고 그 trade-off 를 설명하는 것"이다.
